# AAFC 도메인 전환 시나리오

> 최종 수정: 2026-05-27
> 목적: 기존 ASP 사이트(운영중) → 새 Django(AWS) 사이트로 도메인 전환
> 두 가지 시나리오 동시 수록: **정상(2~4주 준비)** + **긴급(1일 컷오버)**

---

## 0. 핵심 전제

```
[현재]
  www.기존도메인.kr ──→ 기존 ASP/IIS/MSSQL 서버 (운영중)
  (새 도메인 미정)   ──→ AWS EC2 Django (43.201.113.91, http만 가능)

[목표 - 전환 직후]
  www.기존도메인.kr ──→ 기존 ASP 서버 (301 redirect 전용으로 유지)
                            ↓
  www.새도메인.kr   ──→ AWS EC2 Django (https, 실서비스)

[목표 - 1~2개월 후]
  www.기존도메인.kr 종료 (또는 DNS만 새 서버로 이전)
  www.새도메인.kr   ──→ AWS EC2 Django (단독 운영)
```

### 변경되는 핵심 값
| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 서비스 URL | `http://www.기존도메인.kr` | `https://www.새도메인.kr` |
| EC2 접근 URL | `http://43.201.113.91` | `https://www.새도메인.kr` |
| ALLOWED_HOSTS | `43.201.113.91` | `www.새도메인.kr,새도메인.kr` |

| 본인인증 NICE 사이트 등록 | 기존 도메인 | 새 도메인 |
| SMS/이메일 발신 본문 링크 | 기존 도메인 | 새 도메인 |

#### Step F: Nginx 새 도메인 + HTTPS 설정
```bash
# Nginx server block 추가
sudo nano /etc/nginx/sites-available/aafc
```

```nginx
# (HTTP - HTTPS 강제 리다이렉트)
server {
    listen 80;
    server_name www.새도메인.kr 새도메인.kr;
    return 301 https://www.새도메인.kr$request_uri;
}

# (HTTPS - 실서비스)
server {
    listen 443 ssl http2;
    server_name www.새도메인.kr 새도메인.kr;

    ssl_certificate     /etc/letsencrypt/live/www.새도메인.kr/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/www.새도메인.kr/privkey.pem;

    client_max_body_size 50M;

    location /static/ { alias /srv/aafc/staticfiles/; }

    location / {
        proxy_pass http://unix:/run/gunicorn.sock;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # /ba_office IP 제한 (기존 그대로 유지)
    location /ba_office/ {
        allow 1.2.3.4;          # 사무실 IP
        deny all;
        proxy_pass http://unix:/run/gunicorn.sock;
    }
}
```

```bash

# Nginx 적용

# Gunicorn 재시작
sudo systemctl restart gunicorn
```

#### Step G: prod.py 보안 설정 활성화
이미 git push 되어 있다면 자동 적용됨. 안 되어 있다면:
```python
# config/settings/prod.py 주석 해제
SECURE_SSL_REDIRECT = True
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True
SECURE_HSTS_SECONDS = 31536000
SECURE_PROXY_SSL_HEADER = ('HTTP_X_FORWARDED_PROTO', 'https')
CSRF_TRUSTED_ORIGINS = ['https://www.새도메인.kr', 'https://새도메인.kr']
```

#### Step H: 기존 ASP 서버 301 redirect 설정
운영 담당자 또는 IIS 관리자가 진행. `web.config` 최상위에 추가:
```xml
<system.webServer>
  <rewrite>
    <rules>
      <rule name="Redirect to new domain" stopProcessing="true">
        <match url="(.*)" />
        <action type="Redirect"
                url="https://www.새도메인.kr/{R:1}"
                redirectType="Permanent" />
      </rule>
    </rules>
  </rewrite>
</system.webServer>
```

## 8. 자주 묻는 질문

**Q. 회원 비밀번호도 다시 설정해야 하나?**
A. 아니오. `LegacySHA256Hasher`로 기존 SHA256 해시를 그대로 검증합니다. 회원은 기존 비밀번호로 그대로 로그인 가능.

**Q. 기존 도메인 즐겨찾기/북마크는?**
A. 기존 ASP 서버의 301 redirect 덕분에 자동으로 새 도메인으로 이동합니다. 사용자는 한 번 들어왔다가 자동 이동되는 경험.

**Q. 기존에 SMS로 보낸 결제 링크/알림 링크는?**
A. 기존 도메인 URL이라도 301 redirect로 새 도메인으로 자동 이동. **이게 ASP 서버를 한동안 살려두는 이유.**

**Q. 도메인 변경 시 검색엔진 노출 떨어지나?**
A. 301 영구 이동을 쓰면 구글이 점진적으로 새 도메인으로 인덱스를 옮김. 단기적으로는 약간 떨어질 수 있으나 1~3개월 내 회복. Search Console에 새 도메인 등록 + 사이트맵 제출 권장.

**Q. KCP 결제 키를 운영용으로 바꿨는데 테스트에서는 어떻게 확인?**
A. 100원 결제 후 즉시 취소(KCP 가맹점 관리자에서) 가능. 또는 KCP에서 "테스트 카드번호"를 받아 사용.

**Q. DNS 전파 시간(TTL)은?**
A. 평소에 24시간으로 설정되어 있다면, 전환 1~2일 전에 미리 5~10분으로 낮춰두기. 전환 직후 다시 1시간 이상으로 복귀.
