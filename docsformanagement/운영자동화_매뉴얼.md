# AAFC 운영 자동화 매뉴얼

> 원칙: **감지·진단은 자동(agent), 돈·데이터를 바꾸는 결정은 사람.**
> 정형적 90%는 자동화에 위임하고, 사람의 주의는 **"돈 · 동시성 · 피크 시각"** 에 집중.

---

## 1. 4칸 규칙 — 무엇을 어디까지 자동화하나

### 🟢 자동 조치 OK (멱등 + 범위한정 + 횟수제한 + 사후보고)
| 신호 | 자동 조치 | 안전장치 |
|---|---|---|
| gunicorn/nginx 다운 | 재시작 | 5분 내 3회↑ 반복이면 멈추고 사람 호출 |
| 디스크 85%↑ | 오래된 **로그** 압축/삭제 | media/DB는 **절대** 손대지 않음 |
| SSL 인증서 만료 임박 | certbot 자동갱신 + nginx reload | 실패 시 알림 |

### 🟡 알림만 (agent는 진단·제안, 실행은 사람)
| 신호 | agent가 하는 일 |
|---|---|
| 어제 결제 0건 / 결제 실패율 급증 | 로그 분석 → "Toss 키·연동 확인" 제안 |
| RDS 스토리지 80%↑ | "용량 증설 필요" + 증설 명령 준비 |
| 느린 쿼리 · DB 연결 고갈 | 원인 쿼리 짚어 보고 |
| 새 유형 500 에러 | 발생 위치·재현 조건 요약 |
| 트래픽 비정상 폭주 | 출처·패턴 보고 |

### 🔴 절대 자동화 금지 (반드시 사람 승인)
DB 데이터 변경/복원/롤백 · `.env`/키/시크릿 변경 · 코드 배포·마이그레이션 · 결제 취소/환불 · 회원데이터 삭제 · DNS 변경
→ agent는 **"진단 + 제안 + 실행할 명령어 준비"** 까지만. 방아쇠는 사람.

## 2. 감지 대상 (무엇을 본다)
- **가용성**: `https://aafc.co.kr` 200 / gunicorn·nginx active
- **리소스**: 디스크 %, 메모리, RDS CPU·스토리지·연결수
- **데이터 신호**: 주문·수강신청·회원 **일일 증감** (0이면 결제/가입 중단 의심)
- **에러**: Django 500 (Sentry), gunicorn 에러 로그
- **외부**: Toss 결제 성공률, SMS 발송 성공률
- **인증서**: SSL 만료일 (D-30 경고)

## 3. 일일 점검 봇 (매일 아침 자동 리포트)
□ 헬스체크 200  □ 주요 테이블 전일 대비 증감  □ 전일 에러 요약  □ 백업 성공 여부  □ SSL 만료일  □ 디스크/RDS 용량
→ "이상 없음" 또는 "⚠️ 주문 0건 / 디스크 88%" 형태로 통보.

## 4. ⭐ 수강신청 피크 대비 (이 서비스 최대 위험구간)
평소엔 한가하다 **수강신청 오픈·학기 시작에 폭주** → 동시성+결제가 한꺼번에 터짐.
- **피크 D-1**: 수동 백업 1개, RDS 모니터링 강화, 헬스체크 간격 단축(1분)
- **피크 중**: `/loop`로 분 단위 모니터링 — 정원 초과·중복 등록·결제 실패 집중 관찰
- **피크 직후**: 정합성 점검 — ① 정원 초과 등록 없나 ② 결제 ↔ 수강신청 1:1 맞나 ③ 재고 음수 없나

## 6. 정형 사고 1차 대응 플레이북 (빠른 참조)
| 증상 | 1차 확인 → 조치 |
|---|---|
| 사이트 안 열림 | gunicorn·nginx 상태 → 재시작 → 안되면 로그 → 롤백 검토 |
| 결제 안 됨 | Toss 상태·키 확인 → 결제 로그 → (키 문제면 **사람**이 수정) |
| 디스크 풀 | 로그 정리(media 제외) → 안되면 볼륨 증설 |
| DB 느림/멈춤 | 느린 쿼리·연결수 확인 → 필요시 gunicorn 재시작 |
| 데이터 이상 의심 | **자동으로 손대지 말 것** → 진단 보고 → **사람**이 백업 시점 복원 결정 |
